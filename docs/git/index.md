# Git 完全指南

## 简介

欢迎来到 Git 完全指南！本章节将从零开始，系统地介绍 Git 版本控制工具的使用。

## 为什么学习 Git？

Git 是目前世界上最先进的**分布式版本控制系统**，无论是个人开发还是团队协作，Git 都是必不可少的工具。

### Git 的核心优势

- ✅ **版本管理**：完整记录项目的每一次修改
- ✅ **分支管理**：多人协作互不干扰
- ✅ **代码回溯**：随时回到历史版本
- ✅ **远程协作**：团队同步代码
- ✅ **开源贡献**：参与 GitHub 开源项目

## ❓ 常见问题

**Q: 需要有编程基础吗？**
A: 不需要！Git 是通用的版本控制工具，任何开发者都需要掌握。

**Q: Git 和 GitHub 有什么区别？**
A: 很多人容易混淆：
- **Git** 是版本控制工具（软件），安装在本地电脑上
- **GitHub** 是代码托管平台（网站），基于 Git
- 类似关系：Git ≈ 电脑，GitHub ≈ 网盘

**Q: git merge 和 git rebase 有什么区别？**
A: 两者都用于合并分支，但有区别：
- **merge**：保留完整历史，产生合并提交，安全但历史复杂
- **rebase**：线性历史，整洁清晰，但会改写历史（谨慎使用）
- 推荐规则：个人分支用 rebase，公共分支用 merge
- 第 5 章详细讲解 rebase 高级用法

**Q: 如何撤销已经提交的代码？**
A: 根据情况选择：
- **撤销工作区改动**：`git checkout -- file` 或 `git restore file`
- **撤销暂存区**：`git reset HEAD file`
- **撤销上次提交**：`git commit --amend`（改内容）或 `git reset --soft HEAD~`（完全撤销）
- **已推送的提交**：不要撤销，用新提交修复
- 第 2 章完整讲解回退操作

**Q: 冲突了怎么解决？**
A: 按步骤操作：
1. Git 会标记冲突文件（<<<<<<< HEAD）
2. 打开文件，选择保留哪部分代码
3. 删除冲突标记（=======, >>>>>>>）
4. `git add` 标记解决
5. `git commit` 完成合并
- 第 3 章有完整实例演示

**Q: 提交信息怎么写才规范？**
A: 推荐使用约定式提交：
```
feat: 添加用户登录功能
fix: 修复导航栏显示问题
docs: 更新 README 文档
```
- 格式：`类型: 简短描述`
- 常见类型：feat, fix, docs, style, refactor, test, chore
- 第 4 章有完整提交规范

**Q: .gitignore 文件有什么用？**
A: 指定 Git 忽略的文件：
- `node_modules/` - 依赖包
- `.env` - 环境变量（含密钥）
- `dist/` - 构建产物
- `*.log` - 日志文件
- 第 1 章讲解 .gitignore 配置

**Q: SSH 和 HTTPS 连接有什么区别？**
A: 两种方式连接 GitHub：
- **HTTPS**：简单，每次推送要输密码/Token
- **SSH**：配置后免密，更安全
- 推荐使用 SSH，第 1 章有配置步骤

**Q: 如何删除不需要的文件？**
A: 三种情况：
- **未跟踪文件**：直接删除或 `git clean -f`
- **已跟踪文件**：`git rm file`（删除并暂存）
- **保留文件但移出版本控制**：`git rm --cached file`
- 第 2 章有详细说明

**Q: 如何处理大文件？**
A: Git 不适合大文件：
- 单文件建议 < 50MB
- 大文件用 Git LFS（Large File Storage）
- 或使用外部存储（OSS、CDN）
- `node_modules` 等大目录应该加入 .gitignore

**Q: 分支命名有什么规范？**
A: 推荐的分支策略：
- **main/master**：主分支，保持稳定
- **develop**：开发分支
- **feature/功能名**：功能分支
- **bugfix/问题描述**：修复分支
- **hotfix/紧急问题**：紧急修复分支
- 第 4 章讲解 Git 工作流

**Q: 如何查看某次提交的详细内容？**
A: 多种查看方式：
- `git show <commit-id>` - 查看某次提交详情
- `git log -p` - 查看历史和改动
- `git diff <commit1> <commit2>` - 对比两次提交
- `git blame file` - 查看每行代码是谁写的

**Q: 如何恢复误删的分支？**
A: 只要没清理 Git 垃圾：
- `git branch -d <branch>` 删除分支
- 误删后用 `git reflog` 查找记录
- `git checkout -b <branch> <commit-id>` 恢复
- 第 5 章讲解 reflog 和恢复技巧

**Q: 什么时候用 git stash？**
A: 临时保存工作现场：
- 切换分支前，当前代码还没写完
- 拉取远程代码前，本地有冲突
- 临时切换到其他分支修复 bug
- 用法：`git stash` → `git stash pop`
- 第 5 章有完整 stash 教程
